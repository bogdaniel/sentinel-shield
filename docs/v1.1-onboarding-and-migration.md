# v1.1.0 — Onboarding & Migration

> **Additive minor release.** `v1.1.0` adds opt-in capabilities on top of `v1.0.0` and changes **no
> STABLE contract** ([`product-contract.md`](product-contract.md) §1–§3). Upgrading is drop-in.

## Migrate v1.0.0 → v1.1.0 (drop-in)

1. Bump `SENTINEL_SHIELD_REF` to `v1.1.0` (tag or full SHA — never a moving branch).
2. Done. No STABLE surface renamed/removed; exit codes, env vars, schemas, modes, and file modes are
   unchanged. The new transitive Dependency-Check knobs and the extended hardened example are **opt-in
   and default-off**, so existing pipelines behave identically.

What's new in v1.1.0 (all opt-in):
- **Transitive Dependency-Check CI** — `SENTINEL_SHIELD_DEPENDENCY_CHECK_INSTALL_PHP/INSTALL_NODE`
  (default `false`). See [`dependency-check-ci-cache.md`](dependency-check-ci-cache.md).
- **Hardened digest-pinned example** extended with the transitive knobs
  ([`../examples/hardened/sentinel-shield-hardened.snippet.yml`](../examples/hardened/sentinel-shield-hardened.snippet.yml)).
- **Deptrac/IaC promotion plan** (docs only; no maturity change) —
  [`deptrac-iac-promotion-plan.md`](deptrac-iac-promotion-plan.md).

## Production onboarding checklist

```txt
[ ] pin SENTINEL_SHIELD_REF to v1.1.0 (or a full SHA)
[ ] add .sentinel-shield/profile.yaml (start mode: report-only, then baseline)
[ ] add the NVD secret SENTINEL_SHIELD_DEPENDENCY_CHECK_NVD_API_KEY (Dependency-Check)
[ ] pin scanner images/actions by digest/SHA (production) — examples/hardened/
[ ] confirm reports/ is gitignored in the consumer
[ ] run PR-fast on PRs; main-gate on main; Dependency-Check scheduled/nightly (slow)
[ ] keep strict OPT-IN until medium findings + style are triaged
```

## First-project rollout (see also [`consumer-onboarding.md`](consumer-onboarding.md))

1. `report-only` first — observe findings, no gate failures.
2. Move to `baseline` — secrets/critical/high/architecture/type/test gates fire; migration debt tolerated.
3. Triage `medium_vulnerabilities` + `style_violations`; then consider `strict`.

## Multi-project rollout
See [`multi-project-rollout.md`](multi-project-rollout.md). Pin every consumer to the same
`SENTINEL_SHIELD_REF`; bump centrally; immutable tags make the bump reproducible.

## Rollback
- **Sentinel Shield version:** bump `SENTINEL_SHIELD_REF` back to the prior tag (tags are immutable;
  the exact prior behavior is always retrievable).
- **Scanner image digests:** keep the prior `@sha256:` — see [`scanner-image-digest-pinning.md`](scanner-image-digest-pinning.md).
- **Adoption mode:** lower `gates.mode` (e.g. `strict` → `baseline`) in `profile.yaml`.

## Accepted-risk review
Accepted risks are owner + reason + **expiry** ([`accepted-risk-suppression.md`](accepted-risk-suppression.md)).
Review on a cadence: expired records stop suppressing (fail-closed); `secrets` is never suppressible.
A scanner suppression without an exception record is itself a finding.

## Strict-mode opt-in
Strict is **not required by default** — it correctly fails on real findings. Pre-flight in
[`strict-mode-readiness.md`](strict-mode-readiness.md): baseline green + stable, medium/style clean or
accept-risked, style/type gates configured, main-gate scanners live-validated or accepted as advisory.

## Regulated mode is NOT a default
`regulated` adds the strictest gate set (release-evidence, scorecard, the noisier third-party signals).
It is opt-in for teams with that compliance need — never the default.

## What `v1.0.0` (and `v1.1.0`) does NOT mean
- **Not** "every optional scanner is a production default." Most non-core scanner integrations are
  `supported`/`experimental`; their coarse severity is best-effort.
- **Not** a turnkey/zero-config product — adoption needs a profile, pinned refs, and per-project risk
  decisions.
- **Not** an AI-gated or DAST platform — AI review is non-gating; DAST is manual/allowlisted/fail-closed.
- **Not** digest-pinned by default — readable tags for onboarding, digest-pinned overrides for production.
- **Not** transitive-DC by default — opt-in via the v1.1.0 knobs.
