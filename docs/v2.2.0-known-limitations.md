# Sentinel Shield v2.2.0 — Known Limitations

Scope: **engine-only** feature release. This document is the authoritative
limitations statement referenced by the release authorization (GATE 10).

## Not validated in this release

- **Laravel live-consumer validation** — not included. No Laravel profile has been
  exercised against a live consumer repository. `framework-validated` is not claimed.
- **Symfony live-consumer validation** — not included. Same as above.
- **Framework-validated GA** — not authorized (BLOCKED by the release-authorization gate).
- **Full-platform GA** — not authorized (BLOCKED by the release-authorization gate).

## What this release adds (with stated boundaries)

`v2.2.0` extends the engine from a security/release-gate baseline into a broader
engineering-governance baseline. All three gate families are **additive, off by default
in existing modes, and evidence-based** — they judge producer-emitted evidence, they do not
themselves run or understand your tests/architecture:

- **Testing-discipline governance** — TDD **evidence proxies** (production-change-without-test-change,
  changed-line coverage, missing/empty test evidence), BDD executable-specification evidence, and
  ATDD acceptance evidence. It does **not** prove true test-first development, does not guarantee BDD
  quality, and does not replace product-owner acceptance. `missing_behavior_specification` /
  `missing_acceptance_evidence` default on (strict/regulated) for **application profiles only** —
  libraries are never forced to carry BDD/ATDD.
- **Engineering-quality gates** — coverage threshold/regression, mutation score, complexity,
  duplication, dead-code, and missing-coverage-evidence. These gate **producer-emitted metrics**; the
  engine does not compute coverage or run mutation itself.
- **Architecture governance v2** — normalized architecture evidence from Deptrac (PHP),
  dependency-cruiser / ESLint boundaries (JS/TS), or custom producers. Architecture tools detect
  **dependency-boundary violations**, not the quality of domain modeling itself.

Existing mode defaults are unchanged: a project that upgrades and changes nothing sees no new
blocking gate.

## Validated, with stated boundaries

- **Engine** — `engine-tested` via the engine's own default-branch CI (7 required workflows green
  at the release source) and the local production-readiness suite.
- **PHP-library / Node / React consumers** — `standalone-consumer-tested` against controlled
  fixtures, not live production adoption.
- **Adopter harness** — `black-box-adopter-harness-tested`; controlled fixtures, not independent
  production adopters.

## Environment-bound evidence caveats

- **Process-group isolation (bounded-process):** on hosts without `perl`/job control and with
  `SENTINEL_SHIELD_BP_REQUIRE_ISOLATION=1`, the launcher fails closed (refuses to launch) — correct
  defensive behavior. macOS bash 3.2 can mask Linux-CI process-control behavior; CI (ubuntu) is
  authoritative.
- **Live full-scanner security acceptance:** the single end-to-end `security-acceptance.json` over
  the complete production scanner manifest is produced by the authoritative `ci-security` run
  (event=push, branch=master), not reconstructed offline.

## Soak

Not applicable. `v2.2.0` is cut directly from a CI-green default-branch commit; it is not RC-gated.
`verify-candidate` carries no soak gate.
