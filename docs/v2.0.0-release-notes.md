# v2.0.0 Release Notes (DRAFT — engine-only track)

> **Draft.** This is a draft release note for the **engine-only** track of the
> v2.0.0 alpha line. It is **v2.0.0-alpha.1 candidate — not yet published**. No v2
> git tag exists; the highest published tag is `v1.9.2`. Nothing here should be read
> as a shipped, tagged release.

## What changed

v2.0.0 introduces the release-scope governance model, in which a top-level
`release_scope` field (`engine-only`, `framework-validated`, or `full-platform`)
determines which evidence a release must carry. This cycle runs under `engine-only`:
a **production-oriented engine** with **production-readiness controls**, whose
**framework profiles are supported by deterministic tests**. The engine's own
default-branch CI is recorded and GitHub-verified as fail-closed release evidence.

## Who should use it

This release is aimed at teams adopting the engine itself and at maintainers who want
the engine-validated profile behavior. Because it runs under the engine-only scope,
it does not carry live-consumer validation for the framework profiles — see Known
limitations and Deferred validation before adopting for a production Laravel or
Symfony application.

## Upgrade requirements

- The stable line remains v1.x (latest `v1.9.2`); v2.0.0 alpha is a candidate, not a
  published release.
- Review the `release_scope` model and confirm the engine-only scope matches your
  expectations before adopting.
- Migration and profile guidance for v2 lives alongside the existing v2 migration
  documentation in this repository.

## Known limitations

- **Laravel and Symfony profiles have deterministic engine/fixture coverage but were
  not independently validated in real consumer repositories for this release.**
- The release runs under `engine-only`, so it **cannot** claim framework-validated
  status; the validator prints `FRAMEWORK LIVE-VALIDATION NOT INCLUDED`.
- Engine fixtures are synthetic test inputs, not real consumers. Passing fixture
  validation is not production-proof for any framework.
- PHP-library, Node/React, and combined framework profiles are supported but are not
  live-validated unless separately validated.

## Deferred validation

The following are explicitly deferred to future release cycles:

- Real Laravel consumer CI.
- Real Symfony consumer CI.
- Independent production-project adoption proof.
- External maintainer usability proof.

Adopting framework-validated or full-platform scope in a later cycle would require
the corresponding consumer live-validation evidence at beta and above.

## Rollback procedure

- The stable, published line is v1.x (latest `v1.9.2`). Because no v2 tag is
  published, rolling back means staying on or returning to the v1.x line.
- Install/sync/migration operations are transactional with defined recovery
  behavior; a failed engine operation is designed to fail closed and recover rather
  than leave a partial state.
- Consumer-side rollback flows (`rollback_npm`, `rollback_pnpm`, `rollback_yarn`) are
  part of the framework-validated/full-platform GA evidence set and are **not**
  live-validated under this engine-only cycle.
