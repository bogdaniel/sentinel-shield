## Framework live-validation excluded (read first)

**beta.2 does NOT add real-consumer live-validation for the Laravel or Symfony
framework profiles.** Those profiles remain **engine- and fixture-tested only**;
they were **not** run against real Laravel/Symfony consumer repositories this
cycle. The release runs under the `engine-only` release scope, so it **cannot**
claim framework-validated status, and `check-release-readiness.sh` /
`validate-release-evidence.sh` continue to treat framework live-validation as
**deferred**. Nothing in these notes should be read as production proof for a
Laravel or Symfony application. See [Known limitations](#known-limitations),
[Deferred validation](#deferred-validation), and
[`product-status.md`](product-status.md) (canonical maturity).

## What changed in beta.2

beta.2 is a **production-readiness hardening** increment on top of the engine-only
beta.1 pre-release. It is additive: no STABLE engine CLI, exit code, env var, or
schema was renamed or removed. The changes group as follows.

### CI / actions hygiene

- **New workflow-runtime audit gate.** `scripts/audits/workflow-runtime-audit.sh`
  checks every workflow for SHA-pinned actions, least-privilege `permissions`,
  step/job `timeout-minutes`, `concurrency`, and `if-no-files-found` on artifact
  uploads. It is wired into `ci-workflow-lint` (alongside the existing pin/lint
  jobs) so drift fails CI.
- **Action inventory.** A machine-readable inventory of the actions the engine's
  own workflows depend on (see [`workflow-action-inventory.md`](workflow-action-inventory.md)).

### Scanners (versions, metadata, provenance, health)

- **Scanner action bumps, re-pinned by SHA.** OSV-Scanner action
  `v1.9.0` → **`v2.3.8`**; Grype (`anchore/scan-action`) `v4` → **`v7.4.0`**. Both
  are pinned by commit SHA with a `# vX.Y.Z` comment in `ci-security.yml`.
- **Scanner metadata.** Collector output now carries **scanner-version** and
  **db-timestamp** metadata, plus **tool-provenance checksums**.
- **Scanner-health vocabulary.** A normalized health state —
  `ok | findings | no-targets | scanner-error | parser-error` — with a
  **fail-closed exit 2 on `parser-error`** (unparseable scanner output is never
  treated as clean). This is a security-relevant behavior: an unreadable report
  fails the run rather than passing silently.

### Consumer validation (standalone consumers)

- **One unified consumer-validation record schema**
  (`schemas/consumer-validation.schema.json`), emitted through the shared reporter
  library `scripts/report-consumer-validation.sh`, so every consumer produces
  uniform, machine-checkable evidence.
- **Node service + React app — LIVE-TESTED.** `tests/consumers/node-service`,
  `node-*` package-manager variants, and `react-app` run with **real
  npm / pnpm / yarn lockfiles** and real installs
  (`npm ci`, `pnpm --frozen-lockfile`, `yarn --immutable`). Mutations to managed
  files are caught, and rollback is **byte-for-byte**.
- **Standalone PHP library — STRUCTURAL only.** `tests/consumers/php-library`
  validates structure (composer manifest shape, file layout). The **live
  composer / phpunit / phpstan / pint gates are CI-deferred SKIPs** because
  `composer`/`php` are absent in the CI runner — they are **not proven locally**
  this cycle. This is intentionally conservative: do not read the PHP-library
  consumer as live-tooling-validated.

### Installer / recovery hardening

- **Shared transaction library.** `scripts/lib/transaction.sh` was **extracted
  verbatim** from three previously-triplicated inline copies in
  `install-baseline.sh`, `sync-baseline.sh`, and `migrate-v1.sh`; behavior is
  preserved, the logic now lives in one place.
- **Transaction journal + operator recovery.** An append-only transaction journal
  plus `scripts/recover-operation.sh` (`inspect` and recovery modes) let an
  operator inspect and resume the **same fail-closed rollback** the install/sync/
  migrate scripts expose. A tampered/partial journal is rejected (exit 4).
- **Source verification.** `scripts/lib/source-verification.sh` adds optional,
  opt-in **tree-content checksum** and **signed-annotated-tag** verification on top
  of the existing commit-identity `--verify`.
- **Deterministic installation plan.** Installs emit a deterministic
  installation-plan JSON.

### Automation interface (opt-in machine output)

- **`--output json` command-result envelope.** Opt-in on **`doctor`,
  `install-baseline`, `sync-baseline`, `plan-upgrade`, `bootstrap-profile-tools`,
  `run-local-pipeline`, `check-release-readiness`**. Human-readable output and
  exit codes are **unchanged** — the envelope is a no-op unless the exact token
  `--output json` is passed. The envelope conforms to
  `schemas/command-result.schema.json` and is redacted: **secrets are masked and
  absolute local paths are removed** before serialization.
- **Black-box adopter harness.** `tests/adopter/black-box-install.sh` exercises the
  engine the way an external adopter would, from outside the repo.

  > Note: `run-local-pipeline.sh` also has a **separate, pre-existing**
  > `--output-dir <path>` flag controlling the evidence directory. That is not the
  > machine-output envelope; only `--output json` selects the envelope.

### Governance

- **Required-checks registry + audits.** `config/required-checks.json` plus
  `scripts/audits/required-checks-audit.sh` and
  `scripts/audits/merge-safety-audit.sh`, both wired into `ci-workflow-lint`, so
  the set of blocking checks and merge-safety expectations are audited in CI.

### Evidence / provenance (release engineering)

- **Evidence generator.** `scripts/collect-release-evidence.sh` generates a
  candidate `engine_ci[]` evidence document from the GitHub Actions API for one
  exact engine commit (fail-closed; `consumer_runs[]` empty, `required_evidence`
  all false). This complements the existing
  `validate-release-evidence.sh`, which **verifies** an evidence file.
- **Artifact verification.** `scripts/verify-release-artifacts.sh` downloads each
  run's CI artifacts into isolated temp dirs and fail-closed verifies ownership,
  expiration, and **archive safety** (rejects path traversal, escaping symlinks,
  duplicate paths, zip-bombs) via `scripts/lib/archive-safety.sh`.
- **Reproducible release manifest.** `scripts/generate-release-manifest.sh` emits a
  canonical release fingerprint (source commit + tree hash, tag target, scope, CI
  runs, artifact digests, action pins, tool/profile/schema digests) whose
  `reproducibility.hash` is stable across regeneration;
  `scripts/verify-release-manifest.sh` proves self-consistency and (with
  `--evidence`) reconstruction from the same inputs.
- **Finite two-commit finalization.** `scripts/finalize-release-evidence.sh`
  computes one tag target from the two-commit model (`engine_commit` /
  `release_commit`) and creates the tag **only** with `--execute`.
- **Signing / attestation remain deferred** (optional for beta.2).

## Breaking / migration-relevant changes

beta.2 is **additive**; there is **no STABLE contract break** from beta.1. The
items most likely to affect an adopter's automation:

- **Fail-closed `parser-error` (exit 2).** If a scanner emits output that cannot be
  parsed, the run now **fails** (exit 2) instead of proceeding. A pipeline that was
  silently tolerating a broken scanner report will now surface it. This is a
  correctness fix, not a regression.
- **`--output json` is opt-in.** Existing invocations are unaffected. Adopters who
  want machine output should read the envelope from
  `schemas/command-result.schema.json` rather than screen-scraping human output.
- **Scanner action versions moved** (OSV `v2.3.8`, Grype `v7.4.0`). Consumers who
  copied the engine's workflow templates should re-pin to the new SHAs; consumers
  who reference their own pins are unaffected.

See [`migration-beta1-to-beta2.md`](migration-beta1-to-beta2.md) for the step-by-step
upgrade, rollback, and known-limitations detail.

## Known limitations

- **Laravel and Symfony profiles were NOT live-validated** in real consumer
  repositories this cycle (engine + fixture coverage only).
- **The standalone PHP-library consumer is STRUCTURAL only**; its live
  composer/phpunit/phpstan/pint gates are CI-deferred SKIPs, **not proven locally**.
- **Signing and attestation are deferred** (optional for beta.2); release
  provenance is checksum/manifest-based, not cryptographically signed.
- The release runs under `engine-only` scope and **cannot** claim
  framework-validated status.
- Engine fixtures are synthetic inputs; passing fixture validation is not
  production proof for any framework.

## Deferred validation

Explicitly deferred to future release cycles:

- Real Laravel consumer CI live-validation.
- Real Symfony consumer CI live-validation.
- Live composer/phpunit/phpstan/pint execution for the standalone PHP-library
  consumer (currently CI-deferred SKIPs).
- Release signing / build attestation.
- Independent third-party production-adoption proof and external-maintainer
  usability proof.

Adopting framework-validated or full-platform scope in a later cycle would require
the corresponding consumer live-validation evidence at beta and above.

## Rollback

- The stable line is v1.x (latest `v1.9.2`). Because `v2.0.0-beta.2` is a
  pre-release, rolling back means returning to `v2.0.0-beta.1` or to the v1.x line.
- Install / sync / migration operations are transactional and fail-closed; a failed
  operation retains its lock + snapshots and can be inspected and resumed with
  `scripts/recover-operation.sh` (see the migration guide).
